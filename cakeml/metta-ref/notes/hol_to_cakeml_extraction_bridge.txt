The checked theorem import_export_eval_m1_rec states that exported recursive
evaluator results can be imported back to exactly the HOL evaluator result
list. This does not yet verify compiled CakeML execution. It is the first
representation-level checkpoint needed before connecting extracted evaluator
fragments to executable CakeML code.

The checked return-fragment theorem is the first executable-fragment
checkpoint: eval_return_fragment agrees with eval_m1_rec on the return branch,
and the exported fragment imports back to the same result list. The SML and
CakeML prototypes contain the corresponding eval_return_fragment and
export_eval_return_fragment functions, with tests that exercise
import-after-export on a return atom.

The checked add-fragment theorem is the second executable-fragment checkpoint:
eval_add_fragment agrees with eval_m1_rec on the evaluated integer-addition
branch, and exported add-fragment results import back to the same result list.
The SML and CakeML prototypes contain corresponding eval_add_fragment and
export_eval_add_fragment functions, with tests that exercise evaluated
arguments before import-after-export.

The checked eval, chain, and case fragments are the next executable-fragment
checkpoints:

- eval_eval_fragment agrees with eval_m1_rec on the recursive eval branch.
- eval_chain_fragment agrees with eval_m1_rec on the chain branch.
- eval_case_fragment agrees with eval_m1_rec on a simple case branch.

Each exported fragment imports back to the same result list, and the SML and

Bridge plan
-----------

1. Keep the HOL evaluator as the semantic source of truth.
2. Define a small CakeML-compatible core syntax for exported atoms, bindings,
   result lists, and finite host-call profiles.
3. Prove import-after-export round trips for each representation component.
4. Prove that exported evaluator fragments commute with import/export.
5. For pure helpers, use HOL definitions plus the CakeML translator to obtain
   certificate theorems and app specs.
6. For handwritten wrappers or future stateful/FFI code, use CF proofs against
   the HOL specification.
7. Use the existing SML/CakeML/CeTTa oracle tests as regression checks, not as
   substitutes for the HOL bridge theorem.

Bridge policy
-------------

The working policy for the full evaluator bridge is:

1. Pure semantic kernels should be HOL definitions with CakeML translator
   certificates and app-spec certificates.
2. CF should be used for small handwritten wrappers, public surface/string
   boundaries, stateful runner code, and FFI boundaries.
3. Python and shell tooling may lint manifests or consume generated artifacts,
   but must not be treated as semantic authority.

Checked instances of this policy now include:

- Pure helpers translated from HOL: match payloads, chain payloads, source and
  prototype parser token helpers including bounded entrypoints, surface
  primitive value-list wrappers, eval/evalc
  post-processing helpers, and let* first-step payload construction.
- CF wrappers over translated helpers: return/match wrappers, parser wrappers,
  primitive value-list wrappers, and the eval/evalc surface helper wrappers.
- Recursive branch bridge theorems that keep semantic authority in HOL:
  bridge_eval_evalc_fragment_payload_recursive_sound exposes evalc's checked
  value payload and type/error result, while
  bridge_eval_m1_rec_fuller_evalc_payload_sound lifts that payload through the
  common recursive result/family soundness layer.
  bridge_eval_m1_rec_fuller_let_star_payload_sound does the same for the
  successful first-binding let* step, consuming the residual payload theorem
  and the common recursive soundness theorems together.
  bridge_eval_m1_rec_supported_nested_payload_sound now gives one common
  payload-result predicate for the supported nested branch slices: chain,
  evalc, case, switch, and successful-head let*. The let* residual pair
  payload is factored through bridge_let_star_payload_result_sound so future
  let* recursion proofs can consume a named payload predicate rather than a
  large inline existential.
  bridge_eval_m1_rec_supported_nested_fuller_sound consumes that common
  payload predicate and packages it with eval_m1_rec_result_sound,
  eval_m1_rec_nested_family_sound, and eval_m1_rec_family_sound, giving later
  branch proofs one bridge-side theorem to consume for the supported nested
  branch cluster.
  bridge_eval_m1_rec_supported_control_fuller_sound repeats that pattern for
  the control branch cluster consisting of function, return, and eval. Its
  bridge_control_payload_result predicate records function return-payload
  extraction, return identity, and eval's recursive-or-NotReducible result
  behavior alongside the ordinary result/control/family soundness facts.
  bridge_eval_m1_rec_supported_branch_fuller_sound then combines the supported
  nested and supported control clusters into one theorem for later evaluator
  bridge proofs. It deliberately covers only those verified clusters and leaves
  fallback/equality/match/state/FFI behavior outside the combined predicate.
  bridge_eval_m1_rec_fallback_fuller_sound adds a separate, conservative
  bridge predicate for fallback paths: non-proven branches are related to
  eval_m1_ext membership and classified as identity, stack-overflow marker,
  equality step, builtin step, or match-space output. The follow-on theorem
  bridge_eval_m1_rec_supported_or_fallback_fuller_sound combines that fallback
  classification with the supported branch cluster theorem without claiming
  full state, parser, module, or FFI verification.
  bridge_eval_m1_rec_match_fallback_fuller_sound strengthens the match
  subfamily of fallback specifically: `(match &self pattern templ)` results
  are tied to bridge_match_space_payload membership, while still carrying the
  ordinary result/family soundness and generic fallback classification.
  bridge_eval_m1_rec_equality_fallback_fuller_sound does the same for equality
  fallback where eval_m1_ext dispatches to eval_equalities: it records
  eval_equalities membership plus equality_step provenance, while retaining the
  generic fallback classification and result/family soundness facts.
  bridge_eval_m1_rec_builtin_fallback_fuller_sound splits out builtin fallback
  results where eval_m1_ext dispatches through builtin_eval: it records
  builtin_step provenance in addition to the generic fallback classification
  and result/family soundness facts. This is still a pure builtin-evaluator
  theorem, not a full external-call or FFI semantics theorem.
  bridge_eval_m1_rec_identity_fallback_fuller_sound splits out the default
  identity fallback where eval_m1_ext has no match, builtin, or equality result:
  it records that the returned value is the original atom, while retaining the
  generic fallback classification and result/family soundness facts.
  bridge_eval_m1_rec_split_fallback_fuller_sound packages the four fallback
  subfamilies into one bridge theorem for later recursive soundness proofs:
  every non-proven fallback atom is partitioned into match, builtin,
  equality-rewrite, or identity fallback and paired with the corresponding
  subfamily payload witness.
  bridge_eval_m1_rec_supported_or_split_fallback_fuller_sound then combines
  the proven recursive branch clusters with this split fallback theorem, so
  later evaluator bridge proofs can consume one payload predicate for both
  supported branch results and all non-proven fallback results.
  metta_m1_bridge_eval_soundScript.sml starts the downstream modular bridge
  layer: bridge_eval_m1_rec_evaluator_core_payload_sound consumes the combined
  theorem without adding more proofs to the large CakeML bridge theory. The
  positive return and negative add examples document that primitive proven
  branches remain outside this evaluator-core payload predicate until a
  primitive-branch bridge is added.
  The same downstream theory now adds that primitive-branch bridge for the
  arithmetic/comparison and boolean families:
  bridge_eval_m1_rec_primitive_payload_sound records input-sensitive primitive
  payload evidence, and bridge_eval_m1_rec_evaluator_payload_sound combines
  primitive payloads with evaluator-core payloads for later evaluator proofs.
  bridge_eval_m1_rec_model_evaluator_payload_sound is the current total
  HOL-model evaluator payload theorem: every atom is partitioned into the
  model's proven family predicate or split fallback predicate, yielding ordinary
  result/family soundness plus the corresponding model payload. This still does
  not verify the parser, runner state, filesystem/module behavior, or FFI.
- CF branch-facing postconditions:
  bridge_surface_eval_payload_result_wrapper_hand_eval_m1_rec_spec and
  bridge_surface_evalc_checked_values_wrapper_hand_eval_m1_rec_spec state
  app-level postconditions directly in terms of eval_m1_rec branch results,
  not only helper-fragment results.

The next whole-evaluator bridge step is to keep translating pure pieces that
do not close over state, filesystem, or host calls, and to place CF only around
the small wrappers that connect those translated pieces to the handwritten
surface/runner code.

Negative example
----------------

The current bridge does not claim:

  compiled CakeML bytes implement eval_m1_rec

That requires the later CakeML compilation/semantics boundary. The current
theorems prove representation preservation for exported HOL results, checked
executable fragments, source/surface return-core refinement, directly
CF-proved return-block payloads, surface match wrapper agreement, and
translated pure chain/match payloads. The fragment inventory is now derived
from the HOL source, and the marked SML/CakeML fragment blocks are
lint-checked against that inventory. The Python lint tool is not a semantic
generator.
